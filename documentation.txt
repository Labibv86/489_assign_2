### Core App Architecture and Lifecycle Control

The initialization and routing of the application are governed by the `MainActivity` (located in `app/src/main/java/com/example/assignment_2/MainActivity.java`) and the global manifest layout engine (located in `app/src/main/AndroidManifest.xml`). The manifest acts as the structural blueprint for the Android runtime, mapping out every accessible component and declaring explicit permissions. Within this file, the `android:usesCleartextTraffic="true"` configuration flag is explicitly set under the `<application>` tag to override Android’s default security policy against unencrypted HTTP connections, ensuring external media streams can resolve. The execution lifecycle begins within `MainActivity.java`, where the `onCreate(Bundle savedInstanceState)` lifecycle method initializes the core layout container. Inside this method, the `setContentView(R.layout.activity_main)` function inflates the root user interface mapped to `app/src/main/res/layout/activity_main.xml`. Following inflation, a `NavigationView` layout combined with an `ActionBarDrawerToggle` hooks into the support action bar to establish the drawer layout navigation framework. This framework listens for menu interactions and uses `getParentFragmentManager().beginTransaction().replace()` inside a custom navigation listener to switch out the active functional fragments within the central `FrameLayout` container.



### The Native Media Stream Infrastructure

The `VideoFragment` component (located in `app/src/main/java/com/example/assignment_2/VideoFragment.java`) manages the native video playback layout layer, executing inside a container defined by the layout resource file (located in `app/src/main/res/layout/fragment_video.xml`). This XML configuration encapsulates a `VideoView` component centered within a `FrameLayout`. When the fragment transitions through its active creation lifecycle, the `onCreateView` method triggers, inflating the layout and referencing the UI component using `findViewById(R.id.video_view)`. An instance of `MediaController` is instantiated programmatically and bound to the video view using `setAnchorView(videoView)` and `setMediaController(mediaController)` to provide hardware-integrated playback controls. To eliminate network buffering bottlenecks and platform codec errors common on high-API emulators, the data source is loaded locally via a formatted string pointer pointing directly to the compiled resource directory (located in `app/src/main/res/raw/sample_video.mp4`). This URI is passed into `videoView.setVideoURI(Uri.parse(videoPath))`. Playback is synchronized using an asynchronous listener, `videoView.setOnPreparedListener(mp -> videoView.start())`, which delays execution until the media header is fully parsed, ensuring a non-blocking UI rendering thread.



### The Audio Playback Module

Complementing the video engine, the `AudioFragment` (located in `app/src/main/java/com/example/assignment_2/AudioFragment.java`) acts as the dedicated playback manager for localized tracking audio formats. The graphical interface is handled by the layout blueprint (located in `app/src/main/res/layout/fragment_audio.xml`), which arranges user interaction buttons linked directly to audio control methods. Programmatically, this fragment relies on the low-level `MediaPlayer` class infrastructure rather than a layout container wrapper. During the initialization phase within `onCreateView`, the engine initializes the audio pipeline directly from the internal compilation resource directory (located in `app/src/main/res/raw/sample_audio.mp3`) using the factory method `MediaPlayer.create(getContext(), R.raw.sample_audio)`. This setup encapsulates the raw file decoding, resource mapping, and audio channel routing automatically. Interaction buttons are configured with `setOnClickListener` triggers that access state check functions on the media player instance, such as `mediaPlayer.isPlaying()`, `mediaPlayer.start()`, and `mediaPlayer.pause()`. Memory allocation is tightly managed by overriding the fragment lifecycle's `onDestroyView()` method, forcing execution of `mediaPlayer.release()` to prevent system audio channel memory leaks when the user switches to a different feature module.


### The Workflow Selector Interface

The asynchronous communications module initiates within the main `BroadcastFragment` (located in `app/src/main/java/com/example/assignment_2/BroadcastFragment.java`), which functions as Screen 1 of the multi-stage assignment workflow. Built on a `LinearLayout` inside the corresponding layout file (located in `app/src/main/res/layout/fragment_broadcast.xml`), it presents a structural `Spinner` dropdown layout populated with predefined execution paths—the custom broadcast system and the system battery event notification tracker. When the user selects an operational path and triggers the submission button, the `btnProceed.setOnClickListener` execution block runs. The method captures the user's explicit selection index using `spinner.getSelectedItemPosition()`. To optimize the conditional flow and clear static code warnings, the conditional logic is compressed into a boolean expression: `args.putBoolean("IS_BATTERY", selectedPosition != 0)`. This flag is packed directly into a data payload `Bundle` object, which is appended to a new instance of `BroadcastInputFragment` using `inputFragment.setArguments(args)`. Finally, the transaction manager substitutes the layout dynamically, utilizing `addToBackStack(null)` so that the Android system tracks the user's navigation route for seamless backtracking.



### The Input Processing and Battery Monitor Node

Acting as Screen 2 of the workflow sequence, the `BroadcastInputFragment` (located in `app/src/main/java/com/example/assignment_2/BroadcastInputFragment.java`) processes incoming arguments to determine its functional state within its UI layout (located in `app/src/main/res/layout/fragment_broadcast_input.xml`). When `isBatteryMode` resolves to true, the UI hides text inputs, changes the header via `tvTitle.setText()`, and programmatically instantiates an internal `BroadcastReceiver` tracking block. This block registers to catch system events matching the `Intent.ACTION_BATTERY_CHANGED` intent filter via `requireContext().registerReceiver()`. Inside its overridden `onReceive` callback, extraction formulas parse integer keys `BatteryManager.EXTRA_LEVEL` and `EXTRA_SCALE` to dynamically format and render the device's battery metrics, relying on system dictionary definitions mapped out in the resource file (located in `app/src/main/res/values/strings.xml`). Conversely, when set to text input mode, the interface displays an `EditText` field. Upon clicking the execution button, the input text is validated via `messageToSend.isEmpty()`, throwing an input error if null. The method then crafts a secure, application-scoped implicit broadcast intent assigned with a unique identifier token (`com.example.assignment_2.CUSTOM_INTENT_ALERT`). The payload is bundled using `intent.putExtra("PAYLOAD_TEXT", messageToSend)` and scoped explicitly using `intent.setPackage()`. The signal is dispatched through `requireContext().sendBroadcast(intent)`, and navigation is routed back home by invoking `getParentFragmentManager().popBackStack()`.



### The Protected Custom Broadcast Receiver Engine

The foundational processing layer of the communication architecture is maintained by the `BroadcastDisplayFragment` (located in `app/src/main/java/com/example/assignment_2/BroadcastDisplayFragment.java`), which handles the security registration protocols required by modern Android versions. While the updated layout pipeline navigates back to the root view immediately upon broadcast dispatch, this class houses the isolated inner class `BroadcastReceiver customReceiver` that defines the custom receiver engine. The execution parameters bind to a visual component display layout (located in `app/src/main/res/layout/fragment_broadcast_display.xml`). The `onReceive(Context context, Intent intent)` method serves as the main entrance point where incoming system intents are validated against the constant `CUSTOM_ACTION` filter string. Once authenticated, the method reads the message payload using `intent.getStringExtra("PAYLOAD_TEXT")` and outputs the message cleanly via `String.format()` formatting to clear localization layout warnings. Security enforcement takes place during fragment construction, where `androidx.core.content.ContextCompat.registerReceiver` binds the application context to the custom receiver object. By explicitly appending the `ContextCompat.RECEIVER_NOT_EXPORTED` system attribute flag, the registration pipeline enforces OS-level sandboxing, blocking external phone applications from spoofing or capturing the transit data vectors.

### Declaration of AI Usage

Building this native Android application was a challenging process, and I have faced several technical problems during development. In the beginning, I wanted to choose the right framework for the project. When I learned that Flutter does not support the OS level broadcast and Jetpack natively without complex bridge code, then I took help from Google Gemini and it helped me by guiding me to build a native Android app using Java. This shift allowed me to directly access the core system features.

One of the major issues I faced was with video playback. Originally, I tried to stream the video from an online web link. However, the video screen stayed completely black, and the player would not start. I discovered that modern Android versions block unencrypted network traffic by default, which instantly rejected my connection. Furthermore, the virtual network bridge inside the emulator throttled the speed, causing severe buffering lag. To solve this, I moved the downloaded media files directly into the raw resource directory of the project. This local file method completely bypassed the network layers, making the video load instantly.

Another problem appeared when I was working with user interface text. Android Studio kept throwing lint errors about hardcoded string literals and string concatenation. The warnings stated that my text could not be translated easily. I solved this by removing all raw text from my Java files and registering them as resource keys inside the strings file. I also used proper placeholders for the dynamic battery numbers, which satisfied the compiler.

Finally, I struggled with screen navigation logic. When a user sent a custom broadcast message, the application interface remained stuck on the input page. I wanted the application to automatically go back to the main dashboard right after submission. I modified the button click listener to fire off the global intent signal and immediately invoke the fragment manager to pop the back stack. This change smoothly routed the navigation framework back to the home view, creating a much better user experience.
